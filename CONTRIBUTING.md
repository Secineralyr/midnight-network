# Contribution Guide

まずはこのプロジェクトのコントリビュートに興味を持っていただきありがとうございます！💖
ここでは、このプロジェクトに関するコントリビュート方法について抑えておくべき点がまとめられています。

NOTE: Secineralyrにおける全プロジェクトはこのContribution Guideを継承しています。
各プロジェクト別に抑えておくべき点に関しては[プロジェクト別抑えておくべき点](#プロジェクト別抑えておくべき点)セクションを御覧ください！

## 📝開発を行うために

プロジェクト内で何かバグを見つけたり、新たな機能を追加したいときには、必ずissueを作成し、その実装が本当に必要かどうかのレビューをしてもうようにしてください！
この段階を通さずに修正や機能追加をPRしてしまった場合、プロジェクトの本来の仕様や方針から外れる可能性があり、そのPRがマージされないことがあります！

細かい修正に関してはissueを作らずともそのままPRを送って構いません！

## 🗨 Issues

Issueを作成する際は、以下のことを気をつけてください。

- Issueを提出する内容かどうか
  Issueではバグや機能追加提案など、プログラム上に関する問題を解決するために使用されます。
  そのため、プロジェクトに関する質問などはDiscussionsを使用してください！
- 提出するIssueの内容に関して過去に同じものが出されていないかどうか
  同じようなissueができてしまうと、その分だけ管理が大変になってしまいます。
  issueを提出する前に同じようなissueがあるかを確認していた抱くようお願いします。

基本的にissueは用途別に別れているissueテンプレートを元に作成します。
用途に合わないテンプレートでissueを建てないでください。
また、issueテンプレート書かれているissueルールに従うようお願いします。

## 🔧Pull Request

Pull Requestはいつでも歓迎しています！✨
Pull Requestを出す前に、これらをご確認ください。

### ⚠ 提出時の注意点

- タイトルは必ず「PRの種類: 簡潔なタイトル」となるようにしてください
  例: feat: 〇〇に対して〇〇をできるように

  結果的にそのPRでは「何をしたのか？」というのを完結にわかるようにしてください。
- PRテンプレートのコメントに従い、項目は埋めるようにしてください
  これにはPRによって解決されたissueやPRでしたことなどが含まれます。
- PR上のコード責任はマージされるまで提出者にあります
  修正が必要な場合の修正やPR上の質問への回答など、PRが終了するまでは行うようお願いします。

### ✅提出後

PR提出後、プロジェクトごとにCIが実行されます。(あれば)
最低限、そのCIはすべて成功することが条件となります。

ただし、やむを得ない場合はその理由をPRに記載してください。

### 💫ステータス (for member)

Assigneesは、そのPRをマージまで持っていくといういわば「担当者」の位置づけを意味します。
そのため、レビューに関係なく、PRの最終的なマージ責任を負います。

LabelにはそのPRがどの分類に入るものなのかをわかりやすくするためのものです。
なので、わかりやすくするため必ずつけてください。

Projectsには現在進行中のプロジェクトを指定します。
基本的に使われることはないかもしれませんが、あれば追加してください。

Milestoneには今の進行度をわかりやすくするため一番近いものを選択してください。

### ⚖ レビュー (for member)

レビューはそのプロジェクト上のコード品質を維持するためのものです。
そのため、いくつかのレビューする際に見ることを記載します。

- 最低条件
  - そのPRの概要と実際のコードでしていることは正しいか？
  - 特殊な条件を除いてCIはすべて成功しているか？
  - コードは他のコードスタイルを崩さないようになっているか？
- セキュリティ
  - 対応したコードに脆弱性があるものが含まれていないか？
  - そのPRによって他の部分に脆弱性が生まれたりしないか？
- パフォーマンス
  - そのコードによってパフォーマンスが低下しないか？
  - 現状は大丈夫でもこの先パフォーマンスの温床になりえないか？
  - 他に良い方法はないか？
- QA - (リリース後/テスト段階)
  - そのコードに対して十分なテストが行えているか？
  - 実際の挙動に関してバグと思われるような動作をしていないか？
  - そのコードは最終的な使用者に対して便利なものか？(便利になるか？)
  - これらすべての項目を適切に満たしているか？

また、良いと思った部分は少しでも良いのでコメントしてあげましょう！

## 🖋 Commit Message

コミットメッセージには仕様書である[Conventional Commits](https://www.conventionalcommits.org/ja/)が適用されます。

コミットメッセージには必ずこの仕様を遵守するようにお願いします。
また、説明文は英語か日本語の2つの言語で入力できます。

### 🔑型説明

- fix
  バグ修正や既存コードの変更など
- feat
  新たな機能追加や既存コードに新たな処理を追加したとき
- enhance
  利便性向上などの新たな処理の追加ではないが結果的に機能が強化されたとき
- build
  ビルドプロセスの変更
- chore
  対応している形に当てはまらないほど雑多な変更
- ci
  CI(主にGitHub Action)のコンフィグ変更など
- docs
  コメントアウトや、関数にドキュメントを入れたときなど
- style
  formatterなどでコードスタイルが変更された場合
- refactor
  大まかに処理としての変更はないが、lintエラーなどでコードを修正した時
- perf
  パフォーマンス上の修正
- test
  ユニットテストなどの追加や修正
- BREAKING CHANGE
  すべての互換性を完全に破壊してメジャーバージョン的に動かさなくてはならない時
  (こちらはメジャーリリース後限定)

## 🗃 Branch (for member)

フォークをせずにレポジトリ内でコードのやり取りを行う時、レポジトリ内のブランチ管理をするために
必ずケースに分けてブランチを分割することが求められます。

- 最終的なリリース -> `main`
- 開発段階 -> `develop`
- PRする機能系 -> `pr/<簡潔に_内容>`
- 計画されている時期リリース開発 -> `project/?`
